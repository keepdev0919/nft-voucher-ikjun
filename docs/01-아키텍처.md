# 아키텍처

## 전체 구조

```
┌──────────────────────────────────────────────────────┐
│                   사용자 브라우저                        │
│                                                      │
│              React 프론트엔드 (조익준)                  │
│                    │              │                  │
│             Web3.js으로       REST API로              │
│            직접 컨트랙트 호출   백엔드 호출              │
└────────────────────┼──────────────┼──────────────────┘
                     ↓              ↓
          ┌──────────────┐  ┌───────────────────────┐
          │ 스마트 컨트랙트│  │  Spring Boot 백엔드    │
          │ (조현+황종훈) │  │      (정재현)          │
          │              │  │                       │
          │ Solidity      │  │  - 사용자/가맹점 관리   │
          │ ERC-721 NFT   │  │  - 바우처 프로그램 목록 │
          │              │  │  - 사용 로그 관리       │
          └──────────────┘  └──────────┬────────────┘
          [Ganache 로컬]               │
          → 추후 Sepolia 테스트넷      ┌┴──────┐  ┌───────┐
                                     │ MySQL │  │ Redis │
                                     └───────┘  └───────┘
```

## 각 레이어 역할

### 스마트 컨트랙트
- 바우처의 발급, 소유권, 사용 처리를 담당
- 한번 기록되면 수정 불가 → 위변조 방지
- ERC-721 표준: 각 바우처는 고유한 NFT 토큰

### 백엔드 (Spring Boot)
- 목록 조회, 사용자 정보, 가맹점 관리 등 빠른 조회가 필요한 것들
- 블록체인에 올리기엔 비용이 크거나 변경이 잦은 데이터

### 프론트엔드 (React)
- MetaMask 연동으로 사용자 지갑 연결
- Web3.js로 스마트 컨트랙트 직접 호출
- 백엔드 REST API 호출

## 원칙: 뭘 블록체인에 올리고 뭘 DB에 올리나

| 데이터 | 저장 위치 | 이유 |
|--------|-----------|------|
| 바우처 소유권 | 블록체인 | 위변조 불가해야 함 |
| 바우처 사용 기록 | 블록체인 | 영구적으로 투명하게 |
| 가맹점 목록 | DB | 자주 바뀌고 빠른 조회 필요 |
| 사용자 닉네임 | DB | 개인정보, 빠른 조회 |
| 바우처 프로그램 상세 | DB + 블록체인 | 핵심 정보만 체인에 |
