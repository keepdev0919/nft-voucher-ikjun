## 1. 발표 내용

### 1.1 서비스 핵심 로직 프로세스

1. 기관/정부에서 NFT 바우처 **발행**
2. 사용자가 바우처 목록에서 자기에게 해당하는 바우처 **신청**
3. 검증 후 사용자 지갑에 NFT **보관**
4. 사용자가 가맹점에서 QR 스캔하여 **사용** — 가맹점에서 사용 가능 여부 검증
5. 검증 완료 시 금액·장소 등 **기록**이 블록체인에 영구 저장

오늘 발표는 **4·5단계 (사용·기록)** 에 초점. ERC-721 표준에 없는 잔액 차감을 어떻게 구현할 것인지를 중점적으로 다룸.

### 1.2 기술 방향 확정

**ERC-721 표준 NFT 기반 + 토큰별 value 상태값 + ValueUpdated 이벤트 추가한 확장형 구조**로 결정.

| 옵션 | 장점 | 단점 |
|------|------|------|
| ERC-20 | 잔액 이동이 표준 기능 | 여러 종류의 바우처를 각각 식별하기 어려움 |
| ERC-721 기본 | 바우처별 소유권 명확 | 잔액 차감 기능 없음 |
| **우리 선택 (ERC-721 확장)** | 소유권 + 잔액 둘 다 다룸 | 표준 밖이라 이벤트·검증을 직접 설계해야 함 |

### 1.3 교수님 이전 피드백에 대한 우리 해결

| 문제 | 우리 해결 |
|------|----------|
| URI/JSON에 value 저장 시 위변조 탐지 불가 | 컨트랙트 상태값 `values[tokenId]`로 관리 |
| 잔액 차감 표준 기능 없음 | `useVoucher / deduct` 커스텀 함수 정의 |
| 잔액 변경 이벤트 직접 설계 필요 | `ValueUpdated` 이벤트 정의 |
| 외부 익스플로러 자동 인식 불가 | 자체 Event Scanner + Verification API |
| value 변경 이유 추적 필요 | RDB 상세 이력 + 그 해시값을 온체인에 기록 |

### 1.4 시스템 아키텍처 (4-Layer)

- **L1 Frontend**: DApp UI · MetaMask 지갑 연동 · 검증 상태 표시
- **L2 Backend**: API 서버 · RDB · metadata JSON 생성 · keccak256 해시 · 컨트랙트 호출
- **L3 Blockchain**: ERC-721 + `values[tokenId]` + `ValueUpdated` 이벤트 (Ganache)
- **L4 Verification ★**: Event Scanner + Integrity Verifier — RDB 해시와 온체인 해시 대조하여 사후 조작 탐지

### 1.5 데이터 저장 방식

IPFS vs Backend RDB 중 **Backend RDB 채택**. 이유: 조회·관리 편의성, 구현 현실성. RDB는 조작 가능하지만 온체인 해시와 비교하면 조작 탐지 가능.

무결성 검증 4가지 상태:

| 상태 | 의미 |
|------|------|
| ✓ VERIFIED | RDB 해시 == 온체인 해시 → 무결성 확인 |
| ✗ MISMATCH | 해시 불일치 → RDB 사후 조작 의심 |
| ⚠ MISSING_DB | 온체인 이벤트는 있으나 RDB 기록 없음 |
| ⚠ MISSING_ONCHAIN | RDB 기록은 있으나 온체인 이벤트 없음 |

### 1.6 팀원별 담당

| 팀원 | 담당 | 작업 |
|------|------|------|
| 조익준 | Frontend | MetaMask 지갑 연동, 검증 결과 표시 페이지 |
| 정재현 | Backend | RDB 설계, 해시 서비스, metadata JSON 생성 |
| 조현 | Solidity | ERC-721 value 확장, ValueUpdated 이벤트 |
| 황종훈 | Blockchain | Event Scanner, Integrity Verifier, Ganache 배포 |

### 1.7 데모 시연

각 파트 통합 연결까지는 아니고 로컬에서 각자 개발 중. 프론트에서 목업 데이터로 간단 시연:
- MetaMask 연결 → 닉네임 입력 → 사용자/가맹점 롤 선택
- (기관 등록은 백엔드에서 특정 지갑 주소 기반, UI는 추후 구현)
- 사용자 흐름: 둘러보기 → 바우처 신청 → 검증 → 사용 시 QR 생성 → 가맹점 인식 → 결제 → 블록체인 기록

## 2. 교수님 Q&A 피드백

### 2.1 피드백 1 — QR 코드 보안 문제 (앞 팀과 동일 지적)

> **교수님**: "여기도 QR 코드가 문제인 것 같다. 사용자가 QR을 보여주는 구조라면, 누군가가 그 QR을 캡처해 가서 다른 가맹점에서 쓸 수 있는 상황이 발생할 수 있다. 이런 걸 막는 게 필요하다."

> **발표자 (조익준)**: "처음에는 가맹점에서 결제하고 잔액이 온체인에 기록되는 것만 생각해서 QR 보안 문제를 인지하지 못했다. 현재 QR에는 바우처 목록·장소·잔액 같은 정보가 들어있는데, 이 부분이 문제 될 수 있어 추가 논의가 필요하다."

> **교수님**: "앞 팀 발표에서 같은 문제를 지적했다. 결국 **'그 사람이 허락을 해줘야 쓰는' 구조가 필요**하다. 그 부분을 체크해 보면 좋겠다."

**연관 문서**: `07-QR검증-보안설계.md` — EIP-712 서명 + 만료시간 도입 설계 이미 작성 완료. 다음 스프린트에 구현 예정.


### 2.2 피드백 2 — 바우처 자격 검증, 백엔드만으로는 부족

> **교수님**: "바우처 신청 시 이 사람이 신청 가능한 사람인지 체크하는 부분은 있나?"

> **발표자 (조익준)**: "예를 들어 '경기도 청년 바우처' 같은 경우 사용자 정보 중 성별·나이를 비교해서 자격이 안 맞으면 결제 안 되도록 구현 예정. 결제 단계가 아니라 바우처 발급 단계에서 검증한다."

> **교수님**: "그 검증을 컨트랙트에서 하나, 백엔드에서 하나?"

> **발표자**: "백엔드에서 한다."

> **교수님**: "그러면 — 우리가 백엔드에서 검증한다고 해도 **해커가 나쁜 마음 먹고 컨트랙트의 '바우처 발행하기'를 직접 호출하면 망가진다.** 스마트 컨트랙트와 상호작용할 수 있는 게 꼭 이 앱만이 아니라 다른 쪽(직접 트랜잭션 등)에서도 가능하다는 점을 고려해서 고민해 달라."

**핵심 통찰**: DApp의 진입점은 우리 프론트만이 아니다. **누구나 컨트랙트를 직접 호출할 수 있다.** 검증을 백엔드에만 두면 우회당함.

## 4. 핵심 교훈 두 줄

1. **QR 보안 = 우리 팀도 같은 지적을 받았다.** 앞 팀 발표를 보고 미리 인지했지만 발표 자료에 반영 못 함. 다음 발표 전까지 구현해서 "지적 → 해결" 사이클을 보여줘야 함.
2. **"백엔드에서 검증"은 DApp에선 부족하다.** 컨트랙트가 진실의 원천이므로 중요한 검증 로직(자격, 권한, 서명)은 컨트랙트에 둬야 함. 백엔드는 보조 역할.
