# 07. QR 검증 보안 설계

> **상태**: 합의된 설계 (2026-05 ~). 팀 백엔드(`team-backend/`)에 EIP-712 메타 트랜잭션 흐름은 이미 구현 완료(`commit 9823ece`, `0aa1686`).
> 남은 작업은 **QR 생성 주체를 가맹점 측으로 전환**하는 프론트엔드 UX 변경이다.

## 1. 배경 — QR 방향 전환

### 1.1 기존 흐름 (deprecated)

```
사용자가 자신의 바우처 화면에 QR을 띄우고 → 가맹점이 스캔
```

문제점:
- 사용자 QR이 단일 정보(`tokenId` + 소유자 지갑)만 담고 결제 의도(금액, 가맹점)와 분리되어 있음
- 가맹점/금액을 사용자 QR이 모르므로 결제 컨텍스트가 백엔드에서 별도로 조립돼야 함
- QR 캡쳐 후 다른 가맹점에서 재사용될 위험

### 1.2 신규 흐름 (교수님 권유, 합의됨)

```
가맹점이 결제 직전 QR을 띄우고 → 사용자가 스캔
```

장점:
- QR 페이로드에 금액 / 가맹점 / 결제ID / 만료시간을 묶어 발행 시점에 고정 가능
- 같은 QR이 다른 가맹점에서 사용될 수 없음 (가맹점 지갑이 페이로드에 박힘)
- 사용자가 의도된 결제만 서명 (가맹점·금액을 명시적으로 보고 승인)

## 2. QR 페이로드 구조

가맹점 단말이 결제 직전 생성하는 QR은 JSON 문자열을 담는다.

```json
{
  "merchantWallet": "0xMerchant...",
  "amount": 15000,
  "paymentId": "uuid-v4",
  "deadline": 1735689600
}
```

| 필드 | 타입 | 의미 |
|------|------|------|
| `merchantWallet` | string | 결제를 받을 실제 가맹점 지갑 (0x 포함 42자) |
| `amount` | number | 결제 금액 (원 단위) |
| `paymentId` | string | 가맹점 단말이 생성한 결제 세션 ID (UUID v4) — 멱등성/추적용 |
| `deadline` | number | UNIX timestamp(초). QR이 유효한 마지막 시각 (예: 발행 + 60초) |

> 이 QR은 **인증 정보가 아니다.** 페이로드 자체는 누구나 만들 수 있다.
> 실제 결제 보안은 ④~⑥ 단계의 EIP-712 서명 + 컨트랙트 검증에 의존한다.

## 3. 전체 결제 시퀀스

```
사용자(소유자)         가맹점 단말           백엔드                       Voucher 컨트랙트
    │                    │                    │                                 │
    │                    │ ① 결제 세션 시작                                     │
    │                    │   (가맹점이 amount 입력)                            │
    │                    │                    │                                 │
    │                    │ ② QR 생성                                            │
    │                    │   {merchantWallet,                                   │
    │                    │    amount, paymentId,                                │
    │                    │    deadline}                                         │
    │                    │                                                      │
    │ ③ 사용자가 QR 스캔  │                                                      │
    │◄───────────────────┤                                                      │
    │                    │                                                      │
    │ ④ 서명할 바우처 선택 후                                                   │
    │   POST /api/vouchers/{id}/use/prepare                                     │
    │   { merchantWallet, amount }                                              │
    ├─────────────────────────────────────────►│                                │
    │                                          │ ⑤ 검증:                       │
    │                                          │   - voucher.owner == JWT.sub  │
    │                                          │   - status == ACTIVE          │
    │                                          │   - merchant.role == MERCHANT │
    │                                          │   - newValue >= 0             │
    │                                          │ chain.useNonce(tokenId) 조회   │
    │                                          ├────────────────────────────────►
    │                                          │◄────────────────────────────────
    │                                          │ canonical JSON 생성            │
    │                                          │ metadataHash =                 │
    │                                          │   keccak256(canonicalJson)     │
    │                                          │ history(PENDING) 저장          │
    │                                          │ EIP-712 데이터 구성:           │
    │                                          │   merchant = backendWallet     │ ← (중요)
    │  UseVoucherPrepareResponse               │                                │
    │  { historyId, metadataHash, nonce,       │                                │
    │    deadline, eip712 }                    │                                │
    │◄─────────────────────────────────────────┤                                │
    │                                                                           │
    │ ⑥ MetaMask eth_signTypedData_v4(eip712)                                  │
    │   → ownerSignature                                                        │
    │                                                                           │
    │ ⑦ POST /api/vouchers/{id}/use                                            │
    │   { historyId, ownerSignature }                                           │
    ├─────────────────────────────────────────►│                                │
    │                                          │ useVoucherByMerchant(          │
    │                                          │   tokenId, amount,             │
    │                                          │   recordCommitmentHash,        │
    │                                          │   deadline, ownerSignature)    │
    │                                          ├───────────────────────────────►│
    │                                          │                                │ ⑧ 컨트랙트 검증:
    │                                          │                                │   - block.timestamp ≤ deadline
    │                                          │                                │   - structHash 재계산
    │                                          │                                │   - ECDSA.recover == owner
    │                                          │                                │   - approvedMerchant[msg.sender]
    │                                          │                                │   - voucherValue ≥ amount
    │                                          │                                │ → emit VoucherUsed
    │                                          │  Receipt(txHash, block)        │
    │                                          │◄───────────────────────────────│
    │                                          │ history.confirmUse() → CONFIRMED
    │  VoucherUseHistoryResponse               │                                │
    │◄─────────────────────────────────────────┤                                │
```

### 3.1 단계별 책임 요약

| 단계 | 주체 | 작업 |
|------|------|------|
| ① | 가맹점 단말 | amount 입력, paymentId/deadline 생성 |
| ② | 가맹점 단말 | QR 렌더링 (JSON.stringify) |
| ③ | 사용자 디바이스 | QR 스캔, JSON 파싱 |
| ④ | 사용자 → 백엔드 | `/api/vouchers/{id}/use/prepare` 호출 |
| ⑤ | 백엔드 | DB·체인 검증, PENDING 이력 저장, EIP-712 구조 반환 |
| ⑥ | 사용자 디바이스 + MetaMask | `eth_signTypedData_v4` 서명 |
| ⑦ | 사용자 → 백엔드 | `/api/vouchers/{id}/use` 호출 |
| ⑧ | 백엔드 → 컨트랙트 | `useVoucherByMerchant` 전송, Receipt 대기, history CONFIRMED |

> 컨트랙트 호출은 **백엔드가 가스를 대납**한다. 사용자는 가스비를 부담하지 않는다 (메타 트랜잭션).

## 4. 보안 모델 핵심 — `merchant` 필드 두 얼굴

> 이 섹션은 가장 헷갈리는 부분이라 별도로 분리해 둔다.

### 4.1 EIP-712 메시지의 `merchant`

`UseVoucherPrepareResponse.eip712.message.merchant` 값은 **백엔드 지갑 주소**다.
실제 가맹점 지갑이 아니다.

**이유** (`VoucherService.java:244-247`, `Voucher.sol:160`):

```solidity
// Voucher.sol:148-167
function useVoucherByMerchant(
    uint256 tokenId, uint256 amount, bytes32 recordCommitmentHash,
    uint256 deadline, bytes memory ownerSignature
) public returns (bool) {
    require(block.timestamp <= deadline, "Voucher: signature expired");

    address user = ownerOf(tokenId);
    uint256 nonce = useNonce[tokenId];
    bytes32 structHash = keccak256(abi.encode(
        USE_VOUCHER_TYPEHASH,
        tokenId, user,
        msg.sender,                  // ← merchant 자리에 msg.sender(= 백엔드) 사용
        amount, recordCommitmentHash, nonce, deadline
    ));
    address signer = ECDSA.recover(_hashTypedDataV4(structHash), ownerSignature);
    require(signer == user, "Voucher: invalid signature");
    ...
}
```

컨트랙트가 `msg.sender`를 그대로 `merchant` 자리에 넣어 `structHash`를 계산한다.
백엔드가 트랜잭션을 보내므로 `msg.sender` = 백엔드 지갑이다. 따라서 사용자가 서명해야 할
`merchant` 필드도 백엔드 지갑이어야 컨트랙트 검증을 통과한다.

**관련 커밋**: `0aa1686 — fix: EIP-712 merchant 필드 백엔드 지갑 주소로 수정`. 초기 구현은
실제 가맹점 지갑을 넣었다가 `Voucher: invalid signature`로 실패했다.

### 4.2 실제 가맹점 식별은 어떻게?

실제 가맹점 정보는 **canonical JSON 영수증** + **metadataHash**로 anchored된다
(`VoucherService.java:333-342`):

```java
String canonicalJson = String.format(
    "{\"amount\":%d,\"deadline\":%d,\"historyId\":%d,\"merchantWallet\":\"%s\","
    + "\"newValue\":%d,\"oldValue\":%d,\"ownerWallet\":\"%s\",\"tokenId\":%d}",
    amount, deadline, historyId, merchantWallet.toLowerCase(),
    newValue, oldValue, ownerWallet.toLowerCase(), tokenId
);
String metadataHash = keccak256(canonicalJson);
```

- canonical JSON은 DB(`voucher_use_history.canonical_json`)에 보관됨 — 감사 추적용
- `metadataHash`는 컨트랙트 `recordCommitmentHash` 자리로 들어가 온체인에 기록됨
- 즉 체인에는 가맹점 지갑 자체는 안 박히지만, 가맹점 정보를 포함한 영수증의 무결성은 보장됨
- 분쟁 시 백엔드 DB의 canonical JSON을 keccak256 해서 온체인 `recordCommitmentHash`와 비교 → 일치하면 진본

### 4.3 정리

| "merchant" | 어디에 박히나 | 누구를 가리키나 |
|------------|--------------|----------------|
| EIP-712 메시지 `merchant` | 사용자 서명 / 온체인 structHash | **백엔드 지갑** |
| canonical JSON `merchantWallet` | DB + (해시로) 온체인 `recordCommitmentHash` | **실제 가맹점 지갑** |
| `VoucherUseHistoryResponse.merchantWallet` | 백엔드 응답 | **실제 가맹점 지갑** |

## 5. 위협 모델

| # | 공격 | 차단 메커니즘 | 비고 |
|---|------|--------------|------|
| 1 | 위조 QR로 사용자 유도 | 사용자는 prepare → 백엔드 검증 → MetaMask에 표시된 metadata(EIP-712) 확인 후 서명 | 사용자가 MetaMask 팝업에서 금액/토큰ID를 확인해야 함 |
| 2 | 가맹점이 amount를 조작 | EIP-712 서명에 amount가 포함됨. 컨트랙트가 structHash 재계산 → 다른 amount로 호출 시 서명 무효 | ✅ |
| 3 | 사용자 서명 캡쳐 후 재사용 (replay) | 컨트랙트 `useNonce[tokenId]++` 후 다음 호출은 nonce 불일치로 실패 | ✅ |
| 4 | deadline 이후 지연 제출 | 컨트랙트 `require(block.timestamp <= deadline)` | ✅ |
| 5 | 미승인 가맹점 결제 | 컨트랙트 `require(approvedMerchant[msg.sender])` — 단, `msg.sender`는 백엔드라 백엔드는 항상 승인 상태로 두어야 함 | ⚠️ 실제 가맹점 승인 여부는 백엔드가 prepare 단계에서 검증 (`role == MERCHANT`) |
| 6 | 다른 사용자의 바우처를 사용 시도 | 컨트랙트 `signer == ownerOf(tokenId)` 검증 | ✅ |
| 7 | 백엔드 키 탈취 | 백엔드 지갑이 컨트랙트 `owner`라 모든 권한 (mint / approveMerchant / use) — 치명적 | `blockchain.privateKey`는 환경변수로 분리, 운영 시 KMS 등 사용 권장 |
| 8 | 동일 historyId로 두 번 use 호출 | 백엔드 `UseStatus != PENDING` 시 `USE_ALREADY_PROCESSED` | ✅ |

### 5.1 사용자가 MetaMask 팝업에서 확인해야 할 것

MetaMask는 EIP-712 메시지를 사람이 읽을 수 있는 형태로 표시한다:

```
domain: Voucher (chainId 1337, verifyingContract 0x...)
primaryType: UseVoucher
message:
  tokenId: 42
  user: 0x90F8...
  merchant: 0xBackend...       ← 백엔드 지갑 (서명 검증용)
  amount: 15000
  recordCommitmentHash: 0x...
  nonce: 0
  deadline: 1735689600
```

사용자에게 가맹점 정보를 정확히 보여주려면 프론트가 **별도 UI**(canonical JSON에 있는
`merchantWallet`을 백엔드 prepare 응답에서 추출해서 표시)로 보강해야 한다. EIP-712 팝업의
`merchant` 필드를 사용자에게 그대로 보여주면 백엔드 지갑이 표시되어 혼란을 야기한다.

## 7. 운영 시 주의사항

1. **백엔드 지갑 보호 최우선** — 컨트랙트 owner이면서 가스 대납자다. 키 노출 시 임의 민팅·승인 가능.
2. **컨트랙트 재배포 후 정렬** — `truffle migrate --reset` 시 `blockchain.contractAddress`도 갱신해야 EIP-712 `verifyingContract`가 일치한다. 안 맞으면 모든 서명이 무효 처리됨.
3. **체인 ID 일관성** — EIP-712 domain의 `chainId`는 `web3.eth.getChainId()` 결과를 그대로 사용한다. MetaMask 네트워크 chainId(1337)와 일치해야 함.
4. **canonical JSON 변경 금지** — 키 순서, 공백, lowercase 정책을 바꾸면 `metadataHash`가 달라져 과거 영수증 검증이 깨진다.
