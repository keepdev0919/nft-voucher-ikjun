# UX 흐름 및 E2E 시나리오

## 전체 사용자 유형

| Role | 설명 | 가입 방식 |
|------|------|----------|
| 사용자 | 바우처를 신청하고 사용하는 일반 유저 | 회원가입 시 직접 선택 |
| 가맹점 | QR 스캔으로 바우처 결제를 처리하는 업체 | 회원가입 시 직접 선택 |
| 기관 | 바우처 프로그램을 생성하는 주체 | UI 선택 없음 — DB에서 특정 지갑 주소의 role을 직접 수정 |

## 1. 공통 — 최초 진입 및 회원가입

```
앱 진입 (localhost:3000)
    ↓
[로그인 화면]
MetaMask 연결 버튼 클릭
    ↓
MetaMask 팝업 → 계정 선택 → 연결 승인
    ↓
신규 유저 여부 확인 (GET /user/check)
    ├── 기존 유저 → role에 맞는 홈으로 이동
    └── 신규 유저 → 회원가입 진행
            ↓
        [역할 선택]
        사용자 / 가맹점 중 선택
            ↓
        ┌── 사용자 선택
        │       닉네임 입력
        │       POST /user/register { nickname }
        │       → 사용자 홈으로 이동
        │
        └── 가맹점 선택
                닉네임 입력
                카테고리 선택 (일반 음식점 / 영화관)
                POST /merchant/register { nickname, category }
                → 가맹점 홈으로 이동
```

## 2. 사용자 플로우

### 화면 구조
```
사용자 홈
    ├── 전체 바우처 목록   ← 기관이 등록한 바우처 프로그램 전체
    ├── 내 바우처          ← 내가 신청한 바우처만
    ├── 바우처 상세
    └── QR 스캔 (결제)     ← 가맹점 QR 스캔
```

### 상세 흐름

```
[사용자 홈]
    ↓
[전체 바우처 목록]
    - 기관이 등록한 바우처 프로그램 리스트 표시
    - 각 항목: 프로그램 이름, 금액, 유효기간, 카테고리
    - 신청 버튼 클릭
        ↓
    즉시 NFT 민팅 (백엔드가 mintVoucher 대납)
    → 내 바우처에 추가
    → "신청 완료" 토스트 메시지

[내 바우처]
    - 신청한 바우처 목록 (활성 / 사용완료 구분)
    - 바우처 클릭
        ↓
[바우처 상세]
    - 프로그램 이름
    - 잔여 금액 (부분 사용 가능)
    - 유효기간 / 카테고리
    - "이 바우처로 결제하기" 버튼

[QR 스캔 (결제)]
    - 카메라로 가맹점 QR 스캔
    - QR 페이로드: { merchantWallet, amount, paymentId, deadline }
        ↓
    POST /api/vouchers/{id}/use/prepare
    { merchantWallet, amount }
        ↓
    MetaMask 팝업 (EIP-712 UseVoucher 서명)
        ↓
    POST /api/vouchers/{id}/use
        ↓
    [결제 완료] 잔여 금액 / 사용 시각 표시
```

> QR 방향 변경: 사용자는 더 이상 자신의 바우처 QR을 띄우지 않는다.
> 사용자는 항상 **가맹점이 띄운 QR을 스캔**한다. 상세 설계는 `07-QR검증-보안설계.md`.

### 사용자 API 목록

| 기능 | Method | Endpoint |
|------|--------|----------|
| 활성 바우처 프로그램 목록 | GET | `/api/voucher-programs` |
| 바우처 신청 (즉시 민팅) | POST | `/api/vouchers` |
| 내 바우처 목록 | GET | `/api/vouchers/my/{walletAddress}` |
| 바우처 사용 준비 (서명 데이터 요청) | POST | `/api/vouchers/{id}/use/prepare` |
| 바우처 사용 실행 (서명 전송) | POST | `/api/vouchers/{id}/use` |

## 3. 가맹점 플로우

### 화면 구조
```
가맹점 홈
    ├── 정산 대기 금액 합계 (바우처 결제 누적)
    ├── 결제 QR 생성     ← 가맹점이 띄우는 결제 QR
    └── 결제 내역
```

### 상세 흐름

```
[가맹점 홈]
    - 정산 대기 금액 합계 표시
    - "결제 QR 생성" 버튼 / 결제 내역 버튼
        ↓
[결제 QR 생성]
    - 결제 금액 입력 (원)
    - "QR 생성" 클릭
        ↓
    QR 페이로드 생성 (클라이언트):
    {
      "merchantWallet": "<로그인된 가맹점 지갑>",
      "amount": <입력 금액>,
      "paymentId": "<UUID v4>",
      "deadline": <Date.now()/1000 + 60>
    }
        ↓
    화면에 QR 코드 표시 + 60초 카운트다운
        ↓
    사용자가 스캔 후 결제 진행 (사용자 플로우 §2 참조)
        ↓
    백엔드가 useVoucherByMerchant 트랜잭션 confirm 후
    → 가맹점 화면에 "결제 완료" 알림 (폴링 또는 WebSocket)
        ↓
[결제 내역]
    - 날짜별 결제 목록
    - 건별 금액 / 사용자 닉네임 / tx 링크
```

### 가맹점 API 목록 (현행)

| 기능 | Method | Endpoint |
|------|--------|----------|
| 가맹점 회원가입 | POST | `/api/members/merchant` |
| (관리자) 가맹점 온체인 승인 | POST | `/api/members/merchant/{walletAddress}/approve` |
| 결제 준비 (가맹점 단말 주도 — 보조 경로) | POST | `/api/merchant/vouchers/use/prepare` |

> 결제 정산/내역 API는 백엔드 구현 예정. 현재는 `voucher_use_history` 테이블 직접 조회로 대체 가능.

## 4. 기관 플로우 (MVP)

### 화면 구조
```
기관 홈
    ├── 등록한 프로그램 목록
    └── 프로그램 생성
```

### 상세 흐름

```
[기관 홈]
    - 기관이 등록한 바우처 프로그램 목록
    - 프로그램 생성 버튼
        ↓
[프로그램 생성]
    입력 항목:
    - 프로그램 이름
    - 바우처 금액 (원)
    - 총 발행 수량
    - 유효기간
    - 카테고리 (일반 음식점 / 영화관)
    생성 버튼
        ↓
    스마트 컨트랙트에 프로그램 등록
    + 백엔드 DB 저장
        ↓
    홈으로 이동 → 목록에 추가된 프로그램 확인
```

### 기관 API 목록

| 기능 | Method | Endpoint |
|------|--------|----------|
| 프로그램 생성 | POST | `/program/create` |
| 내 프로그램 목록 | GET | `/program/list/{walletAddress}` |

## 5. E2E 시나리오 (전체 흐름)

실제 테스트할 때 이 순서로 진행하면 전체 플로우 검증 가능.

```
[Step 1] 기관 계정으로 로그인
    → 바우처 프로그램 생성
      예) "청년 식비 지원" / 50,000원 / 100개 / 일반 음식점

[Step 2] 사용자 계정으로 로그인
    → 전체 바우처 목록에서 "청년 식비 지원" 확인
    → 신청 버튼 → 즉시 민팅
    → 내 바우처에서 확인
    → 바우처 상세 → QR 코드 확인

[Step 3] 가맹점 계정으로 로그인 (일반 음식점 카테고리)
    → "결제 QR 생성" → 금액 15,000원 입력 → QR 표시

[Step 4] 사용자 계정에서 가맹점 QR 스캔
    → 결제 확인 화면에서 사용할 바우처 선택
    → MetaMask EIP-712 서명
    → 백엔드가 useVoucherByMerchant 대납
    → 사용자 잔여 금액: 35,000원
    → 가맹점 화면에 "결제 완료" 알림

[Step 4-b] 카테고리 불일치 테스트 (TODO — 현재 백엔드 미구현)
    → 영화관 가맹점이 결제 QR 생성
    → 음식점 카테고리 바우처로 스캔 시도
    → 백엔드 prepare 단계에서 카테고리 불일치 오류 반환 예정

[Step 5] 사용자 계정으로 다시 확인
    → 내 바우처 → 잔여 금액 35,000원 확인
```
