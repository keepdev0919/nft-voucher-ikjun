# UX 흐름 및 E2E 시나리오

## 전체 사용자 유형

| Role | 설명 | 가입 방식 |
|------|------|----------|
| 사용자 | 바우처를 신청하고 사용하는 일반 유저 | 회원가입 시 직접 선택 |
| 가맹점 | QR 스캔으로 바우처 결제를 처리하는 업체 | 회원가입 시 직접 선택 |
| 기관 | 바우처 프로그램을 생성하는 주체 | UI 선택 없음 — DB에서 특정 지갑 주소의 role을 직접 수정 |

---

## 1. 공통 — 최초 진입 및 회원가입

```
앱 진입 (localhost:3000)
    ↓
[로그인 화면]
MetaMask 연결 버튼 클릭
    ↓
MetaMask 팝업 → 계정 선택 → 연결 승인
    ↓
신규 유저 여부 확인 (GET /user/check)
    ├── 기존 유저 → role에 맞는 홈으로 이동
    └── 신규 유저 → 회원가입 진행
            ↓
        [역할 선택]
        사용자 / 가맹점 중 선택
            ↓
        ┌── 사용자 선택
        │       닉네임 입력
        │       POST /user/register { nickname }
        │       → 사용자 홈으로 이동
        │
        └── 가맹점 선택
                닉네임 입력
                카테고리 선택 (일반 음식점 / 영화관)
                POST /merchant/register { nickname, category }
                → 가맹점 홈으로 이동
```

## 2. 사용자 플로우

### 화면 구조
```
사용자 홈
    ├── 전체 바우처 목록  ← 기관이 등록한 바우처 프로그램 전체
    ├── 내 바우처         ← 내가 신청한 바우처만
    └── 바우처 상세 + QR
```

### 상세 흐름

```
[사용자 홈]
    ↓
[전체 바우처 목록]
    - 기관이 등록한 바우처 프로그램 리스트 표시
    - 각 항목: 프로그램 이름, 금액, 유효기간, 카테고리
    - 신청 버튼 클릭
        ↓
    즉시 NFT 민팅 (스마트 컨트랙트 호출)
    → 내 바우처에 추가
    → "신청 완료" 토스트 메시지

[내 바우처]
    - 신청한 바우처 목록 (활성 / 사용완료 구분)
    - 바우처 클릭
        ↓
[바우처 상세]
    - 프로그램 이름
    - 잔여 금액 (부분 사용 가능)
    - 유효기간
    - 카테고리
    - QR 코드 표시
        ↓
    가맹점에서 QR 스캔 → 결제 처리 → 잔액 차감
```

### 사용자 API 목록

| 기능 | Method | Endpoint |
|------|--------|----------|
| 전체 바우처 프로그램 목록 | GET | `/program/list` |
| 바우처 신청 (즉시 민팅) | POST | `/voucher/apply` |
| 내 바우처 목록 | GET | `/voucher/my/{walletAddress}` |

## 3. 가맹점 플로우

### 화면 구조
```
가맹점 홈
    ├── 정산 대기 금액 합계
    ├── QR 스캔
    └── 결제 내역
```

### 상세 흐름

```
[가맹점 홈]
    - 정산 대기 금액 합계 표시
    - QR 스캔 버튼 / 결제 내역 버튼
        ↓
[QR 스캔]
    - 카메라로 사용자 바우처 QR 인식
        ↓
    카테고리 검증 (백엔드)
        ├── 불일치
        │   예) 음식점 바우처 → 영화관 가맹점에서 스캔
        │   → "이 가맹점에서는 사용할 수 없는 바우처입니다" 오류 표시
        │   → 다시 스캔 화면으로
        │
        └── 일치
                바우처 정보 표시
                - 소유자 닉네임
                - 잔여 금액
                - 유효기간
                사용 금액 입력
                확인 버튼
                    ↓
            MetaMask 트랜잭션 승인
                    ↓
            온체인 잔액 차감 + 결제 기록 저장
                    ↓
            [결제 완료]
                - 처리 금액
                - 사용자 잔여 금액
                - 확인 버튼 → 홈으로

[결제 내역]
    - 날짜별 결제 목록
    - 건별 금액
    - 정산 대기 합계
```

### 가맹점 API 목록

| 기능 | Method | Endpoint |
|------|--------|----------|
| 바우처 QR 검증 | GET | `/merchant/verify/{tokenId}` |
| 결제 처리 + 기록 저장 | POST | `/merchant/use` |
| 결제 내역 조회 | GET | `/merchant/history/{walletAddress}` |

## 4. 기관 플로우 (MVP)

### 화면 구조
```
기관 홈
    ├── 등록한 프로그램 목록
    └── 프로그램 생성
```

### 상세 흐름

```
[기관 홈]
    - 기관이 등록한 바우처 프로그램 목록
    - 프로그램 생성 버튼
        ↓
[프로그램 생성]
    입력 항목:
    - 프로그램 이름
    - 바우처 금액 (원)
    - 총 발행 수량
    - 유효기간
    - 카테고리 (일반 음식점 / 영화관)
    생성 버튼
        ↓
    스마트 컨트랙트에 프로그램 등록
    + 백엔드 DB 저장
        ↓
    홈으로 이동 → 목록에 추가된 프로그램 확인
```

### 기관 API 목록

| 기능 | Method | Endpoint |
|------|--------|----------|
| 프로그램 생성 | POST | `/program/create` |
| 내 프로그램 목록 | GET | `/program/list/{walletAddress}` |

## 5. E2E 시나리오 (전체 흐름)

실제 테스트할 때 이 순서로 진행하면 전체 플로우 검증 가능.

```
[Step 1] 기관 계정으로 로그인
    → 바우처 프로그램 생성
      예) "청년 식비 지원" / 50,000원 / 100개 / 일반 음식점

[Step 2] 사용자 계정으로 로그인
    → 전체 바우처 목록에서 "청년 식비 지원" 확인
    → 신청 버튼 → 즉시 민팅
    → 내 바우처에서 확인
    → 바우처 상세 → QR 코드 확인

[Step 3] 가맹점 계정으로 로그인 (일반 음식점 카테고리)
    → QR 스캔 → 사용자 QR 인식
    → 바우처 정보 확인
    → 사용 금액 입력 (예: 15,000원)
    → 결제 완료 → 온체인 기록
    → 사용자 잔여 금액: 35,000원

[Step 4] 카테고리 불일치 테스트
    → 영화관 가맹점으로 로그인
    → 동일 QR 스캔
    → "이 가맹점에서는 사용할 수 없는 바우처입니다" 오류 확인

[Step 5] 사용자 계정으로 다시 확인
    → 내 바우처 → 잔여 금액 35,000원 확인
```
